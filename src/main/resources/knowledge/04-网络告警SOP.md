# 网络告警处理 SOP

以下为常见网络类告警的标准处置步骤（SOP），研判网络相关告警时请参考。所有步骤均为人工处置建议，Agent 当前只读、不执行任何操作。

## 网络延迟过高（network_latency_high）

含义：主机或链路的网络往返时延（RTT）超过阈值，可能影响业务响应速度。

处置步骤：
1. 通过告警详情确认涉及的源、目标地址与触发阈值。
2. 用 ping / traceroute 定位延迟出现在哪一跳，区分本端、链路还是对端问题。
3. 检查是否有带宽打满、突发流量、限速策略或设备负载过高。
4. 若为跨机房/跨运营商链路抖动，联系网络组确认线路质量。
5. 延迟恢复到阈值以下并稳定后，再将告警置为已解决/已关闭。

升级条件：延迟持续升高、影响核心交易链路、或伴随大量超时，应升级并通知网络负责人。

## 网络丢包（network_packet_loss）

含义：链路出现丢包，通常表现为连接不稳定、重传增多、业务超时。

处置步骤：
1. 确认丢包率和持续时间，判断是偶发还是持续。
2. 分段排查：本机网卡、交换机端口、中间链路、对端。
3. 检查网卡错误计数、双工/速率协商、线缆与光模块状态。
4. 排除环路、广播风暴、设备 CPU 过载等异常。
5. 丢包消失后观察一段时间，确认稳定再关闭告警。

升级条件：丢包持续或影响核心业务，应尽快升级到网络组处理。

## 端口不可达 / 连接失败（port_unreachable）

含义：目标端口探测失败，可能是服务未监听、被防火墙拦截或网络不通。

处置步骤：
1. 确认目标 IP、端口和探测方式。
2. 在源端用 telnet / nc 测试端口连通性。
3. 分别排查：目标服务是否存活并监听、安全组/防火墙是否放行、路由是否可达。
4. 区分是"网络不通"还是"服务没起"，避免误判方向。
5. 恢复连通后确认业务正常，再关闭告警。

升级条件：核心服务端口不可达且短时间无法恢复，应立即升级。

## 带宽使用率过高（bandwidth_high）

含义：链路带宽接近上限，可能导致排队、丢包与延迟。

处置步骤：
1. 确认是哪条链路、上行还是下行打满。
2. 定位占用带宽的主要来源（备份、同步、异常流量、攻击等）。
3. 临时可限流或错峰，长期需评估扩容或优化流量。
4. 带宽回落后确认业务恢复，再关闭告警。

升级条件：带宽长期打满影响业务、或疑似遭受攻击，应升级并联系安全/网络组。
