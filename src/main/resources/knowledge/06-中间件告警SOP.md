# 中间件告警处理 SOP

以下为常见中间件（消息队列、缓存、连接/线程池）类告警的标准处置步骤（SOP）。所有步骤均为人工处置建议，Agent 当前只读、不执行任何操作。

## 消息队列积压（mq_backlog）

含义：消息队列堆积量持续增长，消费跟不上生产，可能导致业务延迟或超时。

处置步骤：
1. 确认队列名、积压条数与增长趋势。
2. 判断是生产突增，还是消费端变慢或宕机。
3. 排查消费者是否存活、是否报错、单条处理耗时是否变长。
4. 必要时临时扩容消费者或提升并发，同时定位处理变慢的根因。
5. 积压消化、消费恢复正常后再关闭告警。

升级条件：积压持续增长且影响核心业务时效，应升级并通知业务负责人。

## 缓存内存 / 命中率异常（redis_memory_high）

含义：Redis 等缓存内存接近上限或命中率骤降，可能触发淘汰、穿透，加大数据库压力。

处置步骤：
1. 确认实例内存使用率、命中率与阈值。
2. 排查是否有大 key、热点 key、未设置过期时间导致内存堆积。
3. 检查是否存在缓存穿透/击穿，导致请求直打数据库。
4. 评估清理无用 key、设置合理过期或扩容内存。
5. 内存与命中率恢复后再关闭告警。

升级条件：缓存不可用导致数据库压力骤增，应立即升级。

## 线程池 / 连接池耗尽（threadpool_exhausted）

含义：应用线程池或连接池被占满，新请求排队甚至被拒绝，接口超时。

处置步骤：
1. 确认是哪个池（业务线程池、数据库连接池、HTTP 连接池）被打满。
2. 排查是否有慢调用、外部依赖变慢、任务堆积导致占用不释放。
3. 检查池大小配置是否合理，是否存在资源泄漏。
4. 临时可扩大池或重启，但必须同步定位阻塞点。
5. 池占用回落、接口恢复后再关闭告警。

升级条件：耗尽导致接口大面积超时，应立即升级并联系研发。

## 应用无响应 / 健康检查失败（app_health_fail）

含义：应用健康检查失败或无响应，可能是进程假死、依赖不可用或资源耗尽。

处置步骤：
1. 确认实例、健康检查项与失败持续时间。
2. 查看进程是否存活、是否 Full GC、是否 OOM。
3. 排查关键依赖（数据库、缓存、下游服务）是否异常。
4. 结合日志定位根因，必要时重启并保留现场（线程栈、堆快照）。
5. 健康检查恢复、业务正常后再关闭告警。

升级条件：核心应用无响应且短时间无法恢复，应立即升级并拉起应急。
