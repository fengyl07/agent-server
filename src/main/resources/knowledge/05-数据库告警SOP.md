# 数据库告警处理 SOP

以下为常见数据库类告警的标准处置步骤（SOP），研判数据库相关告警时请参考。所有步骤均为人工处置建议，Agent 当前只读、不执行任何操作。

## 数据库连接数过高（db_connections_high）

含义：数据库当前连接数接近或超过上限，新连接可能被拒绝，影响业务。

处置步骤：
1. 通过告警详情确认实例、当前连接数与阈值。
2. 查看连接来源，区分是正常业务增长、连接泄漏还是连接池配置过大。
3. 排查是否存在长事务、未释放连接、慢查询占用连接。
4. 必要时临时提升上限或重启异常应用，但需同步定位根因。
5. 连接数回落到安全范围后，再将告警置为已解决/已关闭。

升级条件：连接数打满导致业务大面积报错，应立即升级并通知 DBA。

## 慢查询过多（db_slow_query）

含义：出现大量慢 SQL，拖慢数据库整体性能，可能引发连接堆积。

处置步骤：
1. 确认慢查询数量、耗时分布和涉及的表。
2. 通过慢查询日志定位具体 SQL 和执行计划。
3. 排查缺失索引、全表扫描、不合理的 JOIN 或大结果集。
4. 与研发确认能否加索引或优化 SQL，避免高峰期执行大查询。
5. 慢查询下降、性能恢复后再关闭告警。

升级条件：慢查询导致主库负载过高或阻塞业务，应尽快升级到 DBA/研发。

## 主从复制延迟（db_replication_delay）

含义：从库落后主库过多，读到旧数据、故障切换有丢数据风险。

处置步骤：
1. 确认延迟秒数（如 Seconds_Behind_Master）和持续时间。
2. 排查从库是否有大事务回放、IO/CPU 瓶颈、网络延迟。
3. 检查主库是否有批量写入、大事务导致 binlog 激增。
4. 延迟恢复后确认主从数据一致，再关闭告警。

升级条件：延迟持续增大或影响读一致性/容灾能力，应升级处理。

## 表空间 / 磁盘不足（db_disk_full）

含义：数据库数据文件或表空间接近用满，可能导致写入失败。

处置步骤：
1. 确认实例、数据目录/表空间使用率与阈值。
2. 定位增长最快的库、表或日志（binlog、redo、临时文件）。
3. 清理可安全删除的历史 binlog/日志，或评估扩容。
4. 使用率回落后确认写入正常，再关闭告警。

升级条件：即将写满且影响核心库，应立即升级并联系 DBA。
